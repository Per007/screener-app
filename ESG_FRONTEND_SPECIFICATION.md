# ESG Screening Application - Frontend Interface Specification

## Executive Summary

This specification outlines the frontend interface for an ESG Portfolio Screening application with the following key characteristics:

- **Primary Use Case**: Ongoing monitoring of multiple portfolios with regular screening
- **User Types**: Admin users (full access) and regular users (read-only), both sharing same interface
- **Design Style**: Traditional finance - conservative, table-heavy, professional
- **Platform**: Desktop-first with tablet support
- **Key Feature**: Failed holdings get immediate visibility with detailed context

## 3. Responsive Design Considerations

### 3.1 Tablet Layout (768px - 1024px)

**Navigation:**
- Top navbar remains horizontal
- Tab labels may condense or use icons
- Hamburger menu for smaller tablets

**Tables:**
- Reduce number of visible columns
- Horizontal scroll for overflow
- Priority columns remain visible

**Cards:**
- 2-column grid instead of 4
- Stack on very narrow tablets

### 3.2 Mobile Consideration

While desktop-first, for basic mobile access:

**Navigation:**
- Hamburger menu
- Vertical navigation drawer

**Tables:**
- Card-based layout instead of tables
- Key information only
- Tap to expand full details

**Forms:**
- Full-width inputs
- Larger touch targets

## 8. Technical Stack Recommendations

### 8.1 Frontend Framework
**React 18+** with TypeScript
- Component-based architecture
- Strong typing for data models
- Large ecosystem

### 8.2 Styling
**Tailwind CSS** or **Styled Components**
- Rapid development
- Consistent design system
- Easy customization

**Alternative:** Plain CSS with CSS modules for traditional approach

### 8.3 State Management
**React Context API** (simpler) or **Redux Toolkit** (more complex apps)
- Global user state
- Table configurations
- Cache API responses

### 8.4 Routing
**React Router v6**
- Declarative routing
- Nested routes
- Protected routes for auth

### 8.5 Forms
**React Hook Form** with **Zod** validation
- Performance (uncontrolled inputs)
- Matches backend validation (Zod)
- Great DX

### 8.6 Tables
**TanStack Table (React Table v8)**
- Powerful, headless table component
- Sorting, filtering, pagination built-in
- Customizable

### 8.7 API Client
**Axios** or **Fetch API**
- HTTP interceptors for auth
- Error handling
- Request/response transformation

### 8.8 Date Handling
**date-fns** or **Day.js**
- Lightweight
- Immutable
- Good TypeScript support

### 8.9 File Handling
**PapaParse** for CSV parsing
**XLSX** (SheetJS) for Excel parsing

### 8.10 Export/Reporting
**jsPDF** for PDF generation
**XLSX** for Excel export
**json2csv** for CSV export

## 9. Accessibility Considerations

### 9.1 Keyboard Navigation
- All interactive elements focusable
- Logical tab order
- Skip links for main content
- ESC to close modals
- Arrow keys for table navigation

### 9.2 Screen Reader Support
- Semantic HTML elements
- ARIA labels where needed
- Status announcements for loading/errors
- Table headers properly associated

### 9.3 Visual
- Sufficient color contrast (WCAG AA minimum)
- Don't rely solely on color for status (use icons + text)
- Resizable text
- No information conveyed by color alone

### 9.4 Forms
- Labels associated with inputs
- Error messages announced
- Required fields clearly marked
- Helpful error messages

---

## 10. Performance Optimizations

### 10.1 Code Splitting
- Route-based code splitting
- Lazy load heavy components (report builder)
- Dynamic imports for screening tools

### 10.2 Caching
- Cache API responses (portfolios, companies, criteria sets)
- Invalidate on mutations
- Local storage for user preferences

### 10.3 Pagination
- Server-side pagination for large datasets
- Virtual scrolling for very long tables
- Infinite scroll as alternative

### 10.4 Optimistic Updates
- Update UI immediately on user actions
- Revert if API call fails
- Better perceived performance

### 10.5 Debouncing
- Search inputs
- Filter changes
- Auto-save

---

## 11. Security Considerations

### 11.1 Authentication
- JWT token stored in httpOnly cookie (preferred) or sessionStorage
- Automatic token refresh
- Logout on token expiration

### 11.2 Authorization
- Role-based access control (RBAC)
- Hide admin features from regular users
- API validates permissions server-side

### 11.3 Data Validation
- Validate all user inputs client-side
- Never trust client validation alone
- Sanitize before display (XSS prevention)

### 11.4 HTTPS
- All API calls over HTTPS in production
- Secure cookie flags
